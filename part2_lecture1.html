ブラウザ・HTML・Webページのしくみ
― あの画面は、どう組み立てられているのか

毎日見ているWebページ。その正体は「ただの文字の文書」だと知っていましたか？

はじめに ― 第2部のはじまり

ここから第2部「応用・社会・現代技術編」が始まります。 第1部では、情報・計算・データといったコンピュータ科学の原理を、土台から積み上げてきました。 第2部では、その土台の上に、私たちが日々触れている現実の技術と社会を重ねていきます。 ブラウザ、ファイル、API、セキュリティ、半導体、データセンター、そして社会制度―― 「理論が、いかにして私たちの世界を形づくっているか」を見ていく旅です。

その第一歩としてふさわしいのが、いちばん身近な技術、Webページです。 私たちは毎日、スマホやパソコンで無数のWebページを見ています。 ニュースを読み、買い物をし、動画を観る。けれど、「そのページが、いったいどうやって表示されているのか」を、 考えたことはあるでしょうか。 じつは、その正体を知ると、少し驚くはずです。 あの色とりどりのページの中身は、もとをたどれば「ただのテキスト（文字の文書）」なのです。 この講では、その文書（HTML）の正体、それを彩るCSS、動きを与えるJavaScript、 そしてそれらを読み解いて画面に描き出すブラウザの仕事を、順に解き明かしていきます。

ちなみに、いまあなたが読んでいるこの教材も、すべてHTMLとCSSで作られています。 見出し、段落、囲み記事、コードの表示、配色の切り替え――そのどれもが、本講で学ぶ仕組みの上に成り立っています。 つまりこの講は、「いま自分が見ているもの」そのものの正体を解き明かす回でもあるのです。 読み終えたら、ぜひこのページの"ソース"をのぞいてみてください（最後の寄り道コラムで触れます）。 習ったばかりのタグや仕組みが、実際に使われているのが見つかるはずです。

HTML（意味の印）→ 構造・見た目・動きの分離 → ブラウザの仕事 → 表示までの流れ。 第1部第13講（インターネット）で学んだ「ページが届くまで」の、その続きの物語です。

HTML ― 文書に「意味の印」をつける

Webページの土台になっているのがHTML（HyperText Markup Language）です。 名前のなかの「マークアップ（markup）」が、その本質を表しています。 マークアップとは、文章に「ここは見出し」「ここは段落」「ここは画像」といった"意味の印"をつけていくこと。 もとは、原稿に編集者が「ここは大きな見出しで」と赤字で指示を書き込む作業を指す言葉でした。 HTMLは、それをコンピュータ向けに行う仕組みなのです。

具体的には、タグと呼ばれる目印で文章を挟みます。 たとえば、こんなふうに。

<h1>今日のニュース</h1>        ← これは大見出し
<p>朝から good な天気です。</p>   ← これは段落
<a href="...">続きを読む</a>      ← これはリンク

<h1> で始まり </h1> で終わる部分が「大見出し」、 <p> で挟まれた部分が「段落」、というように、 文章のひとかたまりごとに「これは何なのか」という意味のラベルを貼っていく。 HTMLがやっているのは、見た目を作ることではなく、文書に"意味の構造"を与えることなのです。 ここが最初のポイントです。HTMLは「どう見せるか」ではなく「これは何か」を記述する言語なのです。

そして名前のもうひとつの要素、「ハイパーテキスト（HyperText）」。 これはリンクのことです。文書の中の言葉から、別の文書へ飛べる仕組み。 ふつうの本は最初から順に読みますが、ハイパーテキストは、気になった言葉をたどって、文書から文書へと自由に渡り歩ける。 この「リンクで文書どうしがつながる」という発想こそ、Web（World Wide Web ＝ 世界に張り巡らされた蜘蛛の巣）の名前の由来であり、 その核心です。第1部第8講で学んだグラフ（点と線）を思い出してください。 Webとは、文書を点、リンクを線とする、世界規模の巨大なグラフなのです。

マークアップという発想の深さ

「文章に意味の印をつける」というマークアップの発想は、じつはとても奥が深いものです。 たとえば同じ「重要な言葉」でも、ただ太字にするのと、「これは重要だ」という意味の印（<strong>）をつけるのとでは、まったく違います。 前者は「見た目」の指示にすぎませんが、後者は「意味」の宣言。 意味の印がついていれば、見た目はあとからCSSで自由に変えられるし、読み上げソフトは「ここは強調されている」と声の調子を変えて伝えられます。 「どう見えるか」ではなく「これは何か」を記述する――この一見遠回りな考え方が、 人にも機械にも扱いやすい、柔軟な文書を生むのです。 紙の文書が「見た目がすべて」だったのに対し、HTMLは「意味と見た目を切り離した」点に、革命がありました。

マークアップ言語にはHTML以外にもいろいろあり、たとえばデータ交換によく使われるXMLや、 手軽に書ける記法として人気のMarkdown（このような教材や、チャット、技術文書でよく使われます）も、同じ仲間です。 どれも共通して「素のテキストに、決まった記号で意味の印をつける」という発想に立っています。 プレーンなテキストでありながら、構造を持たせられる―― この「人間にも機械にも読める」というバランスのよさが、マークアップが長く使われ続ける理由なのです。

構造・見た目・動きの分離 ― HTML / CSS / JavaScript

では、あの美しくデザインされたページは、どうやって作られるのでしょうか。 現代のWebページは、役割の違う3つの言語が分業して作っています。 この「役割分担」こそ、Webのいちばん大切な設計思想です。

Webページを作る3つの言語

HTML：文書の構造・意味を決める（これは見出し、これは段落…）。建物でいえば「骨組み」。

CSS：その見た目を決める（色、大きさ、配置、フォント…）。建物でいえば「内装・外装」。

JavaScript：動き・対話を与える（ボタンを押すと反応する、内容が動的に変わる…）。建物でいえば「電気・設備」。

なぜ、わざわざ3つに分けるのでしょうか。それは「構造」と「見た目」と「動き」を分けておくと、片方を変えても、他方に影響しないからです。 たとえば、サイト全体のデザインを変えたいとき、CSSだけ書き換えれば、HTML（中身）はそのままで見た目だけ一新できる。 逆に、記事を追加したいときは、HTMLに文章を足すだけで、デザインは自動的に適用される。 もし、文章とデザインがぐちゃぐちゃに混ざっていたら、ちょっとした修正のたびに全体を作り直す羽目になります。

これは、第1部で何度も登場した「関心の分離」「抽象化」という思想そのものです。 第1部第11講のプログラミング言語でも、第13講のネットワークの階層化でも、 「役割ごとに分けて、互いに干渉しないようにする」という同じ考え方が出てきました。 複雑なものを、独立した部品に分けて管理する―― この知恵が、Webの世界でも、HTML・CSS・JavaScriptという形で貫かれているのです。

CSSの書き方を、少しだけのぞいてみましょう。たとえば、こんなふうに書きます。

h1 { color: red; font-size: 32px; }   ← 見出しは赤の32px
p  { line-height: 1.8; }              ← 段落は行間を広めに

「どの部分（h1やp）を、どう飾るか（色・大きさ・間隔）」を指示しているだけ。 HTMLには一切デザインの情報を書かず、こうしてCSSに見た目をまとめておく。 すると、サイト全体で何百ページあっても、このCSSを1か所変えるだけで、全ページのデザインが一斉に変わります。 第1部第12講のデータベースで学んだ「ひとつの事実は一か所だけに書く（正規化）」と、同じ精神です。 同じことを何度も書かず、一か所にまとめておけば、変更も管理も楽になる―― この原則は、コンピュータのあらゆる場面で繰り返し現れる、普遍的な知恵なのです。

ブラウザの仕事 ― 受け取って、組み立てて、描く

では、これらHTML・CSS・JavaScriptを受け取って、実際に画面に表示するのは誰でしょう。 それがブラウザ（Chrome、Safari、Edge、Firefoxなど）です。 ブラウザは、私たちが思っている以上に、複雑で高度なソフトウェアです。 その仕事は、おおまかに次の流れになります。

  1. 受け取る：Webサーバーから、HTML・CSS・JavaScriptや画像などのファイルを受け取る（第13講の通信）。
  2. 解釈する：HTMLを読んで、文書の構造を木構造（第7講！）に組み立てる。これをDOMと呼ぶ。
  3. 見た目を計算する：CSSを読んで、どの要素を、どんな色・大きさ・位置で表示するかを計算する。
  4. 描く：計算結果にもとづいて、実際に画面にピクセルを描き出す（レンダリング）。
  5. 動かす：JavaScriptを実行し、ユーザーの操作に応じてページを変化させる。

注目したいのは、HTMLが木構造として組み立てられる点です。 「ページ全体の中に本文があり、本文の中に見出しと段落があり…」という入れ子の関係は、まさに第1部第7講で学んだ木そのもの。 この木をDOM（Document Object Model）と呼び、JavaScriptはこの木をたどったり、 枝を付け替えたりすることで、ページを動的に変化させます。 第1部で学んだデータ構造が、ブラウザの心臓部で実際に働いているのです。

ブラウザの中核には、これらの処理を高速に行うレンダリングエンジンと、 JavaScriptを高速に実行するエンジンが組み込まれています。 とくにJavaScriptエンジンは、第1部第11講で学んだ「コンパイラ／インタプリタ」の技術の結晶で、 書かれたコードをその場で機械語に近い形へ変換しながら、猛烈な速さで実行します。 私たちが「Webサイトがサクサク動く」と感じる裏には、こうしたブラウザの地道で高度な最適化があるのです。

サーバーとクライアント ― 役割分担のもうひとつの形

Webの仕組みを理解するうえで、もうひとつ大切な区別があります。 サーバーとクライアントです。 私たちの手元のブラウザが「クライアント（お願いする側）」、 ページのデータを保管して送り出す向こう側のコンピュータが「サーバー（提供する側）」。 レストランにたとえれば、注文する客がクライアント、料理を作って出す厨房がサーバーです。 この「お願いする側」と「提供する側」に分けて、必要なときだけやり取りする仕組み（クライアント・サーバー方式）が、 Webの基本構造です。

そして、処理を「手元（クライアント）でやるか、向こう（サーバー）でやるか」という分担にも、設計の妙があります。 たとえば、入力した内容のかんたんなチェックは手元のJavaScriptで即座に行い、 本当に重要な処理（決済やデータ保存）は、信頼できるサーバー側で行う。 手元の処理は速くて軽快ですが、利用者がいじれてしまうので、最終的な正しさはサーバーが責任を持つ―― この役割分担は、第2部第5講以降のセキュリティの話にも、深くつながっていきます。 「どこで、誰が、何を担うか」を分けて考える――これもまた、第1部以来の「関心の分離」の応用なのです。

ページが表示されるまでの一部始終

ここで、第1部第13講（インターネットの基礎）とつなげて、 URLを入力してからページが表示されるまでの流れを、通して振り返ってみましょう。 これは、本講義の前半で学んだ知識が、すべて連携する場面です。

  1. 名前を引く：入力したURLのドメイン名を、DNSでIPアドレスに変換する（第13講）。
  2. サーバーにつなぐ：そのIPアドレスのサーバーに、TCP/IPで接続する（第13講）。httpsなら暗号の準備も（第10講）。
  3. ページを要求する：「このページをください」とHTTPで要求し、サーバーがHTMLなどを返す。
  4. 組み立てる：ブラウザがHTMLを木（DOM）にし、CSSで見た目を計算し、JavaScriptを動かす（本講）。
  5. 描く：最終的に、画面に色のついたピクセルとして描き出される（第2講の画像＝数値の表！）。

たったコンマ数秒のあいだに、これだけのことが起きています。 情報を数値で表し（第2講）、ネットワークを越えて運び（第13講）、暗号で守り（第10講）、 木構造に組み立て（第7講）、コードを翻訳して実行する（第11講）。 「ページを開く」というありふれた動作のなかに、本講義で学んだほぼすべての要素が詰まっているのです。 第2部の最初にこの話をするのは、まさにそのためです。Webページは、原理と応用が出会う、絶好の入口なのです。

Webには、ほかの技術にはない、ちょっと素敵な文化があります。 それは「ソースを見る（View Source）」――どんなWebページでも、その元になっているHTMLやCSSを、 誰でも自分のブラウザで簡単にのぞけるのです（多くのブラウザで右クリックから「ページのソースを表示」できます）。 つまり、世界中の素晴らしいWebサイトのすべてのHTMLが、いわば"公開された教科書"。 初期のWeb開発者の多くは、憧れのサイトのソースを見て、真似ながら独学で学びました。 優れた作品の作り方が、誰にでも見えるように開かれている―― この「見せ合う・学び合う」精神が、Webを爆発的に発展させた原動力のひとつでした。 バーナーズ＝リーが技術を無償公開したことと合わせ、Webの根っこには「開かれていること」への強い信念が流れているのです。

歴史 ― Webの誕生と「ブラウザ戦争」

たった一人から始まった、World Wide Web。 Webは1989年、イギリスの科学者ティム・バーナーズ＝リーによって考案されました。 当時、彼が働いていた巨大な研究所では、たくさんの研究者が、バラバラの形式で大量の文書を持っていて、 互いの情報を参照するのが大変でした。 そこで彼は、「文書どうしをリンクでつなぎ、どこからでもたどれるようにすればいい」と考え、 HTML・URL・HTTPという、いまもWebを支える3つの基本を一人で設計したのです。 そして決定的だったのは、彼がこの技術を誰でも自由に使えるよう、無償で公開したこと。 特許で囲い込まず、世界に開放したからこそ、Webは爆発的に広がりました（第13講「開かれた約束」の精神です）。

ブラウザ戦争。 Webが普及すると、その入口であるブラウザをめぐって、企業間の激しい競争が起こりました。 1990年代後半から2000年代にかけての「ブラウザ戦争」では、各社が独自機能を競い合った結果、 「このサイトはこのブラウザでしか正しく表示されない」といった混乱も生まれました（第2講の文字コード乱立に似ています）。 その反省から、いまではWeb標準という共通ルールが整えられ、 どのブラウザでも同じように表示される世界へと近づいてきました。 ここでも「みんなで共通の約束を守る」ことの大切さが、繰り返し確認されています。

バーナーズ＝リーが当初こだわったのは、Webが「読む」だけでなく「誰もが書ける」場所であることでした。 一部の発信者だけでなく、すべての人が情報を発信できる――その理想は、ブログやSNSという形で、ある意味で実現しました。 ただし、誰もが発信できることは、玉石混交の情報やデマがあふれることも意味します。 「開かれていること」は、自由と混乱を同時にもたらす。 Webの理想と現実のあいだのこの緊張は、いまも私たちが向き合い続けている課題なのです。

現代への応用

Webページから「Webアプリ」へ

かつてWebは、文書を「読む」だけの場所でした。 しかしJavaScriptの進化により、いまやWeb上で、地図を動かし、表計算をし、動画を編集し、ゲームまで遊べます。 ブラウザの中で、本格的なアプリケーションが動く時代になったのです（Webアプリ）。 メールも、文書作成も、ビデオ会議も、いまやアプリをインストールせず、ブラウザだけでできる。 ブラウザは「文書を見る窓」から「あらゆるアプリが動く土台（プラットフォーム）」へと進化したのです。 この流れは、特定の機種やOSに縛られず「ブラウザさえあればどこでも同じことができる」という、大きな自由をもたらしました。 WindowsでもMacでもスマホでも、同じWebアプリが同じように動く――これは、利用者にとっても作り手にとっても大きな利点です。 作り手は、機種ごとに別々のアプリを作らずに、ひとつのWebアプリを用意すればよい。 「どこでも同じように動く」という約束（Web標準）が、ソフトを"場所から解放した"のです。 いまや、複雑な業務システムから、ちょっとしたツールまで、その多くがWeb上で動いています。 ブラウザは、現代における"もうひとつのOS"とも言える存在になりつつあるのです。

「見た目の分離」が生む、アクセシビリティ

HTMLが「見た目」ではなく「意味の構造」を記述することには、深い意義があります。 目の不自由な人が使う読み上げソフトは、HTMLの「ここは見出し」「ここはリンク」という意味の印を手がかりに、 ページを音声で正しく伝えます。 もし見た目だけで「なんとなく大きい文字」にしていたら、機械にはそれが見出しだと分かりません。 意味をきちんと印づけることが、誰もが情報にアクセスできる「アクセシビリティ」につながるのです。 検索エンジン（第1部第8講）がページの内容を正しく理解できるのも、この意味の構造のおかげ。 「構造と見た目を分ける」という設計思想が、人にも機械にもやさしいWebを作っているのです。

状態を覚える ― Cookieとログインのしくみ

ひとつ、素朴な疑問があります。Webの通信は、基本的に「お願いして、答えをもらう」一回ごとの単発のやり取りで、 サーバーは前回誰が来たかを基本的には覚えていません（ステートレス、と言います）。 では、なぜログインした状態が保たれ、ショッピングカートの中身が消えないのでしょうか。 その鍵がCookie（クッキー）です。 サーバーは、訪問者のブラウザに小さな"合言葉"（Cookie）を渡しておき、 次回以降、ブラウザがそれを提示することで「あ、さっきの人だ」と認識する。 単発のやり取りの上に、「覚えている」状態を人工的に作り出しているのです。 この仕組みのおかげで、私たちは毎回ログインし直さずに済みます。

ただし、この「覚える」仕組みは、便利さと引き換えに、プライバシーの問題も生みました。 Cookieを使えば「この人がどのサイトを見て回ったか」を追跡することもでき、広告に利用されてきました。 近年、こうした追跡への懸念から、Cookieの扱いには世界的に厳しいルールが設けられつつあります （サイトでよく見る「Cookieに同意しますか？」という確認は、その表れです）。 便利さと、プライバシーの保護――この綱引きは、第2部全体を通じて何度も登場する、現代技術の重要なテーマです。

よくある誤解

誤解①：「Webページは、画像のような"絵"として送られてくる」

違います。送られてくるのは主に「テキスト（HTML・CSS・JavaScript）」で、それをブラウザが受け取って、 その場で画面に"描き起こして"います。同じHTMLでも、ブラウザや画面サイズによって見え方が変わるのはこのためです。

誤解②：「HTMLはプログラミング言語だ」

厳密には違います。HTMLは「文書に意味の印をつける」マークアップ言語で、計算や条件分岐（第11講）はしません。 計算や動きを担うのはJavaScriptです。HTMLは"設計図"、JavaScriptが"動き"、と役割が違います。

誤解③：「インターネットとWebは同じもの」

別物です（第13講）。インターネットは「世界中の機器をつなぐ土台」、Webはその上で動く「サービスのひとつ」。 メールや動画通話も、同じインターネットの上を走る、Webとは別のサービスです。

誤解④：「ブラウザはどれも同じで、見た目を映すだけの単純なソフト」

ブラウザは、HTMLの解釈、CSSの計算、JavaScriptの高速実行、セキュリティ管理まで担う、極めて高度なソフトウェアです。 OSに匹敵する複雑さを持つ、とも言われます。

誤解⑤：「ログイン状態は、サーバーが私のことをずっと覚えてくれている」

基本のWeb通信は単発で、サーバーは前回の訪問を自動では覚えていません（ステートレス）。 ログイン状態が保たれるのは、ブラウザが持つCookie（合言葉）を毎回提示しているから。 「覚えている」状態は、この仕組みで人工的に作り出されているのです。

まとめ

  • Webページの正体は「ただのテキスト文書」。それをブラウザが受け取り、その場で画面に描き起こしている。だから同じページでも、ブラウザや画面サイズで見え方が変わる。
  • HTMLは「見た目」でなく「意味の構造」を記述するマークアップ言語。リンクで文書をつなぐ（＝巨大なグラフ）。
  • HTML（構造）・CSS（見た目）・JavaScript（動き）の3分業。「関心の分離」という設計思想。
  • ブラウザはHTMLを木構造（DOM）として組み立て、CSSで見た目を計算し、JavaScriptで動かして、画面に描き出す。
  • ページ表示には、第1部のDNS・TCP/IP・暗号・木構造・コンパイラが総動員される。
  • Webはバーナーズ＝リーが考案し、無償で公開したことで爆発的に普及した。「開かれた標準」と「ソースを見せ合う文化」が、世界を一つにつないだ。

考えてみよう

  1. HTML・CSS・JavaScriptの役割を、それぞれ一言で説明せよ。建物のたとえを使ってもよい。
  2. 「構造（HTML）」と「見た目（CSS）」を分けておくと、どんな利点があるか。1つ挙げよ。
  3. HTMLが「プログラミング言語ではない」と言われるのはなぜか。JavaScriptとの違いから説明せよ。
  4. 「インターネット」と「Web」は、どう違うか。第1部第13講も思い出して説明せよ。
  5. Webページが「巨大なグラフ」だと言えるのはなぜか。第1部第8講のグラフの定義から考えよ。
ヒントと答え

(1) HTML＝構造・意味（骨組み）、CSS＝見た目（内装）、JavaScript＝動き・対話（電気設備）。
(2) 見た目だけを変えたいときCSSだけ直せばよく、中身（HTML）に影響しない。逆に中身を足してもデザインは自動適用される。修正や管理が楽になる。
(3) HTMLは「意味の印をつける」だけで、計算や条件分岐・繰り返しをしない。それらの"動き"を担うのがJavaScript。
(4) インターネットは世界中の機器をつなぐ土台（道路網）、Webはその上で動くサービスの一つ（宅配便）。メール等もインターネット上の別サービス。
(5) 文書を「点（ノード）」、リンクを「線（エッジ）」とみなせるから。Webは点と線でできた世界規模のグラフそのもの。

さらに読む本・資料

  • 『HTML&CSS とウェブデザイン』系の入門書（たとえばジョン・ドゥケット『HTML&CSS』） ― 実際に手を動かして、HTML・CSSの基本を学べる定番のビジュアル入門書。本講の内容を体験できる。
  • 『Web技術の基本』（小林恭平・坂本陽）（SBクリエイティブ） ― HTTP・URL・ブラウザ・サーバーの関係を、図解でやさしく整理した一冊。第13講と本講の橋渡しに。
  • ティム・バーナーズ＝リー『Webの創成』（毎日コミュニケーションズ） ― Web考案者本人が、その発想と理念を語った書。「なぜ無償で公開したのか」という思想が読める。
  • MDN Web Docs（developer.mozilla.org） ― HTML・CSS・JavaScriptの、信頼できる無料のオンライン辞典。さらに深く学ぶときの定番リファレンス。
