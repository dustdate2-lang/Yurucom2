APIとソフトウェア同士の会話
― アプリは、こうして手を組む

地図アプリに天気が出る。決済が一瞬で済む。その裏で、ソフト同士が"会話"しているのです。

はじめに ― ソフトが、別のソフトに「お願い」する

あるアプリの中に、別のサービスの地図が埋め込まれている。 飲食店の予約アプリから、そのまま地図と経路が見られる。 ネット通販で、クレジットカード会社の決済画面が呼び出される。 天気アプリが、気象会社のデータを表示する。 ―― こうした「アプリ同士の連携」は、いまや当たり前の風景です。 では、これらの別々の会社が作ったソフトウェアは、どうやって互いに協力しているのでしょうか。

その答えが、本講のテーマAPI（Application Programming Interface）です。 APIは、ソフトウェアが、別のソフトウェアに「これをやってください」とお願いするための"窓口"。 現代のデジタルサービスは、自分ですべてを作るのではなく、 さまざまな会社のAPIを呼び出し、組み合わせることで成り立っています。 いわば、ソフトウェアの世界の「分業と協力」の仕組みです。 人間社会が、すべてを一人でやらず、専門家どうしが分業することで豊かになったように、 ソフトウェアの世界も、APIによる分業で、爆発的に発展してきました。 地図は地図の専門家に、決済は決済の専門家に任せ、自分は自分の得意なことに集中する―― この当たり前の知恵を、ソフトの世界で実現する仕組みが、APIなのです。 この講では、APIとは何か、ソフト同士はどんな"言葉"で会話するのか（リクエスト・レスポンス・JSON）、 そして「コードを書かずにアプリを作る」ノーコードの世界までを、身近な例から順に解き明かしていきます。 ふだん何気なく使っているアプリの便利さの正体が、きっと見えてくるはずです。 これは第1部第11講で学んだ「抽象化」「インターフェース」の考え方が、現実のサービス連携として花開く回です。

少し大きな視点で言えば、APIは現代のデジタル経済の"配管"のようなものです。 水道や電気が、見えないところで街じゅうに張り巡らされて生活を支えているように、 APIは、見えないところで無数のサービスをつなぎ、私たちの便利な体験を支えています。 ひとつのアプリの操作の裏で、何社ものサービスのAPIが一瞬のうちに呼び出され、結果が組み合わされて画面に表示される。 私たちが「ひとつのアプリ」だと思っているものの正体は、じつは無数のサービスの"合奏"なのです。 この配管がどう敷かれているかを知ると、デジタルサービスの世界の見え方が、まるで変わってきます。 では、その配管をのぞいていきましょう。

APIとは（機能の窓口）→ リクエストとレスポンス → JSON → APIがつなぐ世界 → ノーコード。 「自分で全部作らない」という、現代のものづくりの作法が見えてきます。

APIとは ― 機能を呼び出すための「窓口」

APIを理解する、いちばん分かりやすいたとえはレストランのメニューと注文です。 客（あるソフト）は、厨房（別のソフト）がどんな手順で料理を作っているかを知る必要はありません。 メニュー（API）に書かれた品名を、ウェイター（窓口）に注文すれば、料理（結果）が出てくる。 客は「何を頼めるか」と「どう頼むか」だけを知っていればよく、「どう作っているか」は知らなくていい。 この仕組みこそ、APIの本質です。

API（Application Programming Interface）

ソフトウェアが持つ機能を、外部の別のソフトウェアから呼び出して使うための「窓口（接点）」。 「こう頼めば、こういう結果を返す」という約束（取り決め）として公開される。 利用する側は、内部の作りを知らなくても、その窓口を通じて機能を借りられる。

ここで思い出してほしいのが、第1部第11講・第7講で繰り返し登場した「使い方（インターフェース）と中身（実装）の分離」です。 APIは、まさにこの思想そのもの。 天気予報のAPIを使う側は、気象会社が世界中の観測データをどう集め、どう計算しているかを一切知らなくていい。 ただ「東京の明日の天気をください」とAPIに頼めば、答えが返ってくる。 「中身を隠して、窓口だけを公開する」――この抽象化が、世界中のソフトを部品のように組み合わせ可能にしたのです。 第1部で学んだ抽象化の思想が、企業や国境を越えた、巨大な協力のネットワークを生んでいるのです。

身のまわりにもAPIはあふれている

APIは、なにもインターネット越しの会話に限った話ではありません。 じつは、私たちが普段使うソフトの内側でも、あちこちでAPIが使われています。 たとえば、アプリが「写真へのアクセスを許可しますか？」と聞いてくるとき、 そのアプリは、スマホのOS（基本ソフト）が用意した「カメラやアルバムを使うためのAPI（窓口）」を呼び出そうとしているのです。 アプリが自分でカメラを直接動かすのではなく、OSが用意した窓口を通じて、安全に機能を借りる。 「中身を直接いじらせず、決められた窓口だけを通す」という仕組みは、安全管理の要でもあります。 もし、どのアプリもカメラや連絡先を好き勝手に触れたら、大変なことになりますね。 APIという"窓口"を通すからこそ、「このアプリにはカメラを許可、でも位置情報は不許可」といった、きめ細かい管理ができるのです。

このように、APIには大きく2つの顔があります。 ひとつは、本講のメインである「インターネット越しに、別の会社のサービスを呼び出す」Web API。 もうひとつは、ひとつの機械の中で「OSや他のソフトの機能を借りる」ためのAPI。 どちらも本質は同じで、「中身を隠し、決められた窓口を通じて機能を提供する」という約束ごとです。 この共通の発想を押さえておくと、ニュースで聞く「API連携」も、スマホの「アクセス許可」も、同じ仕組みの表れだと見えてきます。

リクエストとレスポンス ― お願いと、お返事

では、ソフト同士は具体的にどうやって会話するのでしょう。 基本は、第1部第13講・第2部第1講で学んだWebの通信と同じ、「お願いして、お返事をもらう」という形です。

  • リクエスト（要求）：使う側が「これをやって」「このデータをください」と送るお願い。
  • レスポンス（応答）：提供する側が「はい、結果はこれです」と返すお返事。

たとえば、天気のAPIに「東京の今日の天気」というリクエストを送ると、 「晴れ、最高28度」といったレスポンスが返ってくる。 この一往復が、ソフト同士の会話の基本単位です。 多くの場合、このやり取りはWebの仕組み（HTTP、第2部第1講）の上で行われるため、 とくにWeb APIと呼ばれます。 私たちがブラウザでページを「要求して受け取る」のと同じ仕組みを、 人間ではなく、プログラムが、別のプログラムに対して使っているのです。 つまりWeb APIとは「機械が機械に向けて使う、Webの窓口」だと考えると分かりやすいでしょう。

このリクエストには、ただ「お願い」を書くだけでなく、いくつかの大事な要素が含まれます。 「誰からの依頼か（身元を示す合言葉）」「何をしてほしいか」「どんな条件で（東京の、今日の、など）」。 とくに重要なのがAPIキーと呼ばれる合言葉です。 多くのAPIは、誰でも無制限に使えるわけではなく、利用者にあらかじめ専用の鍵（APIキー）を発行し、 リクエストのたびにそれを提示させます。 これによって「誰が、どれだけ使ったか」を把握し、使いすぎを防いだり、料金を計算したりできるのです。 第1部第10講で学んだ「鍵で本人確認をする」発想が、ここでも生きています。

そして、お返事（レスポンス）には、第2部第1講で学んだ状態コードも付いてきます。 「成功（200）」「そんなデータはない（404）」「あなたには権限がない（401）」「サーバー側の不具合（500）」――。 APIを使う側のプログラムは、この番号を見て「うまくいった」「失敗した、ではこう対処しよう」と判断します。 機械同士の会話にも、ちゃんと「了解」「ごめん、できません」といった作法が決められているのです。 うまくいくときの処理だけでなく、失敗したときにどう振る舞うかを考えるのが、API連携の腕の見せどころになります。

JSON ― 機械同士の「共通語」

ソフト同士がデータをやり取りするとき、そのデータの書き方にも、共通のルールが要ります。 人間同士が共通の言語で話すように、機械同士も共通の形式でデータを表す必要がある。 いま、その共通語として最も広く使われているのがJSON（ジェイソン）です。

JSONは、第2部第2講で学んだテキスト形式の一種で、 「項目名」と「値」をペアにして、データを表します。たとえば、こんなふうに。

{
  "都市": "東京",
  "天気": "晴れ",
  "最高気温": 28,
  "降水確率": 10
}

波カッコの中に、「都市は東京」「天気は晴れ」と、項目と値が並んでいる。 人間が読んでも、だいたい意味が分かりますね。これがJSONの良いところです。 機械が効率よく処理でき、かつ人間にもギリギリ読める――この絶妙なバランスが、JSONを普及させました。 第1部第12講の「表形式データ」やCSVと同じく、「データを、決まった形に整えて受け渡す」発想の仲間です。 APIのレスポンスの多くは、このJSONの形でデータを返してきます。 天気でも、地図でも、SNSの投稿でも、機械同士はこうした構造化されたテキストで、淡々と情報を交換しているのです。

かつてはXMLという、よりかっちりした形式も広く使われ、いまも分野によっては現役です。 どの形式を使うにせよ、大切なのは「送り手と受け手が、同じ形式の約束を共有している」こと。 これは第1部第2講の文字コード、第2部第2講のファイルフォーマットと、まったく同じ「解釈の約束」の話。 コンピュータの世界では、この「共通の約束ごと」が、あらゆる連携の土台になっているのです。

JSONのもうひとつの良さは、入れ子（ネスト）にできることです。 値の中に、さらに別のJSONを入れられるので、複雑な構造のデータも表現できます。 たとえば「ある人の情報」の中に「複数の住所」のリストを入れ、その住所のそれぞれに「郵便番号と都市」を入れる、というように。 これは第1部第7講・第8講で学んだ木構造そのもの。 データの親子・包含の関係を、そのまま表現できるのです。 だからJSONは、単純な表（CSV）では表しにくい、複雑で階層的なデータのやり取りに向いています。 「人間が読めるテキスト」でありながら「木構造を表せる」―― この2つを両立させたことが、JSONが現代のデータ交換の主役になった理由なのです。

APIがつなぐ世界 ― 組み合わせて作る時代

APIの本当のすごさは、「他人の作った機能を、自分のサービスに取り込める」点にあります。 地図を表示したいなら、地図会社のAPIを呼べばいい。 決済をしたいなら、決済会社のAPIを使えばいい。 わざわざ自分で世界中の地図を作ったり、銀行と提携したりする必要はありません。 現代のアプリの多くは、こうした他社のAPIを部品のように組み合わせて作られているのです。 複数のサービスを組み合わせて、新しい価値を生むこの作り方を、マッシュアップと呼びます。

これは、ものづくりのあり方そのものを変えました。 かつてソフトを作るには、すべてを自前で開発する必要がありました。 しかしAPIの時代には、「世界中の優れた機能を借りてきて、つなぎ合わせる」ことで、 小さなチームでも、驚くほど高機能なサービスを、短期間で作れるようになった。 これは第1部第11講の「ライブラリ」や「巨人の肩の上に立つ」という話の、サービス全体への拡張版です。 「すべてを自分で作らない」ことが、むしろ賢いものづくりの作法になった―― これが、API時代の大きな転換です。

もちろん、これには光と影があります。 便利な反面、自分のサービスが他社のAPIに依存することにもなる。 頼っていたAPIが、ある日、有料になったり、仕様が変わったり、提供を終了したりすれば、 自分のサービスも影響を受けてしまう。 「他人の機能を借りる」便利さと、「他人に依存する」リスク―― この綱引きは、第1部第11講の「共有の恩恵とリスク」とも通じる、現代のソフトウェアの宿命です。

ノーコード ― コードを書かずに作る

APIの発想をさらに推し進めると、面白い世界が見えてきます。ノーコードです。 これは、プログラムのコードを書かずに、用意された部品を画面上で組み合わせるだけで、アプリやWebサイトを作る仕組み。 ブロックを積むように、「ボタンを置く」「フォームを置く」「このAPIとつなぐ」といった操作を、 マウスでドラッグしながら組み立てていきます。

これは、第1部第11講で見た「抽象化の階段」の、さらに上の段だと考えられます。 機械語 → 高級言語 → そして「コードすら書かず、部品を組み合わせる」へ。 人間が考えなくていいことを、また一段、機械側に肩代わりさせたのです。 ノーコードのおかげで、プログラミングの専門家でない人――現場の事務担当者や、小さな店のオーナーなど――でも、 自分の業務に合った道具を、自分で作れるようになりました。これは大きな民主化です。

ノーコードと相性がいいのが、本講のテーマであるAPIです。 ノーコードのツールは、たいてい「このサービスのAPIと、あのサービスのAPIを、こうつなぐ」という連携を、 コードを書かずに、画面上の線でつなぐだけで設定できます。 「フォームに入力があったら → その内容を表計算に記録し → 担当者にメッセージで通知する」といった自動化を、 部品を並べるだけで組める。 APIという"つなぐための窓口"が整っているからこそ、ノーコードという"つなぐ作業の簡略化"が成り立つのです。 この2つは、「専門知識がなくても、ソフトの力を使えるようにする」という同じ方向を向いた、現代の大きな潮流です。 そして、この潮流の先に、第1部第15講で見た「AIに言葉で頼んでアプリを作ってもらう」という、さらに上の段が続いていきます。

とはいえ、ノーコードには「光」だけでなく「闇」もあります。 手軽に作れる反面、細かい調整がきかない、特定のサービスに縛られる、内部で何が起きているか分からないといった弱点があるのです。 用意された部品の枠を超えたことはできないし、そのノーコードサービス自体が終了すれば、作ったものごと失われかねない（第2部第2講の「独自形式」の問題と同じです）。 また「簡単に作れる」がゆえに、セキュリティや保守の知識がないまま重要なシステムを作ってしまい、後で問題が噴出することもあります。 手軽さの裏には、必ず「見えなくなったもの」がある。 だからこそ、ノーコードを賢く使うにも、その下で何が起きているか――まさに本講義で学ぶような土台の理解――が役に立つのです。

歴史と背景 ― なぜAPIの時代が来たのか

「公開する」という戦略。 APIという考え方自体は古くからありましたが、それが爆発的に重要になったのは、Webサービスの時代です。 ある企業が、自社の機能（地図、決済、SNS投稿など）をAPIとして外部に公開すると、 世界中の開発者が、それを使って新しいサービスを作りはじめます。 すると、その企業のサービスは、自社だけでは思いつかなかった無数の使われ方で広がっていく。 機能を「囲い込む」のではなく「開いて使ってもらう」ことが、結果的に大きな力になる―― これは第2部第1講のWeb、第13講のインターネットで見た「開かれた約束が普及を生む」という法則の、ビジネス版です。 いまや「APIを公開しているか」は、プラットフォームとしての価値を測る重要な指標になっています。

その一方で、APIは企業の戦略の道具にもなります。 多くの開発者が自社APIに依存するようになった後で、突然、利用条件を変えたり有料化したりして、 開発者コミュニティとの間で軋轢が生まれる、という出来事も繰り返し起きてきました。 「開いて広げる」段階と「囲って回収する」段階――プラットフォームをめぐるこの力学は、 第2部の後半（インフラや社会制度の回）で、より大きなスケールで再び登場します。

現代への応用

あなたのスマホは、API連携の塊

気づいていないだけで、私たちは毎日APIのお世話になっています。 食事の宅配アプリは、地図API・決済API・通知APIを組み合わせて動いています。 SNSの「他のアプリでログイン」は、認証のAPI。 スマホの「この予定を、カレンダーに追加」も、アプリ間のAPI連携。 天気、乗換、株価、翻訳――こうした情報が、いろいろなアプリに横断的に現れるのは、 その大元のデータがAPIで提供されているからです。 現代のアプリは、単体で完結する"製品"ではなく、無数のサービスをつなぐ"結節点"になっているのです。 ひとつの宅配アプリを開いている数秒のあいだに、お店の情報、地図、配達員の位置、決済、通知――と、 いくつもの別々のサービスのAPIが呼ばれ、その結果がひとつの画面に滑らかにまとめられている。 裏側を知ると、あの当たり前の便利さが、いかに多くの"協力"の上に成り立っているかが見えてきます。

そしてAIも、APIでつながる

近年は、AI（第1部第15講）もAPIで提供されるのが主流になりました。 高度なAIを一から作れなくても、AI企業が公開するAPIを呼び出せば、 自分のアプリに「文章の要約」「翻訳」「画像の生成」といったAI機能を組み込めます。 さらに、AIが外部のAPIを"自分で呼び出して"情報を取りに行く、という新しい使い方も広がっています （第1部第15講で触れた「AIが道具を使う」の一例です）。 APIは、人間とソフト、ソフトとソフト、そしてAIと世界をつなぐ、共通の"接点"として、 ますます重要になっているのです。 この「つなぐ仕組み」を理解することは、これからのデジタル社会を読み解く、大切な鍵になります。

APIが止まると、世界が止まる

APIによる連携が当たり前になった結果、ひとつの重要なサービスのAPIが止まると、 それに依存していた無数のサービスが、いっせいに不調になるという事態も起きるようになりました。 ある大手クラウドや認証サービスに障害が起きると、まったく無関係に見える多くのアプリやサイトが、同時に使えなくなる。 ニュースで「大規模なネット障害」と報じられる出来事の多くは、こうした"つながりすぎた"ことの裏返しでもあります。 便利につなぐほど、どこか一か所の不調が全体に波及しやすくなる―― これは第1部第13講の「分散の頑健さ」とは逆に、「集中への依存」というもろさです。 効率と頑健さのトレードオフ、特定サービスへの依存の危うさは、第2部のインフラの回（データセンターやNVIDIAの話）で、 さらに大きなスケールで考えることになります。 APIは現代社会の便利な"配管"であると同時に、その配管が詰まったときの影響も、また大きいのです。

よくある誤解

誤解①：「APIを使うには、相手のソフトの中身（仕組み）を理解しないといけない」

逆です。APIの利点は、まさに「中身を知らなくても、窓口を通じて機能を借りられる」こと。 レストランで料理の作り方を知らなくても注文できるのと同じ。これが「インターフェースと実装の分離」の恩恵です。

誤解②：「APIは、エンジニアだけに関係する専門的な話」

私たちが毎日使うアプリの多くが、裏でAPI連携で動いています。アプリ同士が協力できる仕組みを知ることは、 サービスがなぜ便利か、なぜ時々まとめて止まるのか、なぜ「他のアプリでログイン」ができるのか――といった、 日常の素朴な疑問を理解する助けになります。

誤解③：「ノーコードなら、プログラミングの知識はまったく要らない」

作るだけなら不要なこともありますが、トラブル対応・セキュリティ・他サービスとの連携を考えると、 土台の理解はやはり役立ちます。「見えなくなった部分」に何があるかを知る人ほど、ノーコードも上手に使えます。

誤解④：「他社のAPIを使えば、もう何の心配もいらない」

便利な反面、依存も生まれます。頼っていたAPIが有料化・仕様変更・終了すると、自分のサービスも影響を受ける。 「借りる便利さ」と「依存するリスク」は、つねにセットで考える必要があります。

誤解⑤：「APIのやり取りは、JSONのような"特別な技術"でしか行われない」

中身は、第1部・第2部で学んだ基本の組み合わせです。HTTPで通信し（第13講・第1講）、テキスト形式でデータを表し（第2講）、 状態コードで結果を伝える（第1講）。APIは新しい魔法ではなく、既存の仕組みを"機械同士の会話"に使い回したものなのです。

まとめ

  • APIは、ソフトが別のソフトに機能を「お願い」するための窓口。中身を知らずに機能を借りられる。Web越しの連携にも、スマホOSのアクセス許可にも、同じ仕組みが使われている。
  • レストランの注文と同じ。客は「何を・どう頼むか」だけ知ればよく、厨房の「どう作るか」は知らなくていい（インターフェースと実装の分離＝抽象化）。
  • 会話の基本はリクエスト（お願い）とレスポンス（お返事）。APIキー（合言葉）で本人確認し、状態コードで結果を伝える。Webの上で行うものをWeb APIと呼ぶ。
  • データの共通語はJSON（人間も読めるテキスト形式で、木構造も表せる）。「解釈の約束を共有する」点は、文字コードやファイル形式とまったく同じ発想。
  • 他社のAPIを組み合わせて作る（マッシュアップ）のが現代のものづくりの作法。「借りる便利さ」と「依存するリスク」は、つねに表裏一体。
  • ノーコードは抽象化のさらに上の段で、ものづくりの民主化を進める。一方、手軽さの裏には、自由度の制限や特定サービスへの依存など「見えなくなるもの」もある。

考えてみよう

  1. APIを「レストラン」にたとえると、客・メニュー・厨房は、それぞれ何にあたるか。
  2. APIを使う側が「相手の中身を知らなくていい」のは、なぜ便利なのか。第1部第11講の言葉で説明せよ。
  3. JSONが、機械同士のデータ交換の共通語として好まれるのはなぜか。その特徴を挙げよ。
  4. 他社のAPIに依存することの、利点とリスクを、それぞれ1つずつ挙げよ。
  5. ノーコードの「手軽さ」と引き換えに失われやすいものは何か。1つ挙げよ。
ヒントと答え

(1) 客＝使う側のソフト、メニュー＝API（頼めることの一覧）、厨房＝機能を提供する側のソフト。ウェイター＝窓口。
(2) 「インターフェース（使い方）と実装（中身）が分離」されているから。中身を気にせず使えるので、互いに独立して開発でき、組み合わせも自由になる。
(3) テキスト形式で人間にも読め、かつ機械が効率よく処理できる。項目名と値をペアで構造的に表せる。
(4) 利点：自分で作らずに高度な機能を借りられ、開発が速い。リスク：仕様変更・有料化・終了などで自分のサービスが影響を受ける。
(5) 細かい調整の自由、内部で何が起きているかの理解、特定サービスへの非依存（移行のしやすさ）など。

さらに読む本・資料

  • 『Web API: The Good Parts』（水野貴明）（オライリー・ジャパン） ― Web APIの設計と考え方を、平易に解説する定番書。APIの「良い窓口とは何か」が分かる。
  • 『Webを支える技術』（山本陽平）（技術評論社） ― HTTP・URL・REST（Web APIの基本様式）を体系的に学べる名著。第2部第1〜3講を貫いて読める。
  • 『figmaやノーコードツールの公式チュートリアル』など ― ノーコードは、実際に触ってみるのが一番。代表的なツールの無料チュートリアルで、組み合わせる感覚をつかめる。
  • 各サービスの「API ドキュメント（公式リファレンス）」 ― 地図・天気・SNSなど、多くのサービスがAPIの使い方を公開している。「どう頼めば何が返るか」の実例集として面白い。
