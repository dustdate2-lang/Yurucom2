アルゴリズムと計算量
― 「解ける」と「速く解ける」は別物

同じ問題でも、手順しだいで答えが出るのが一瞬にも、宇宙の年齢ぶんかかるようにもなる。

はじめに ― 速さという、もうひとつの物差し

この講を一言でいうと

プログラムやパソコンが「遅い」とき、私たちはつい「スペックが足りないんだ」と思いがちです。でも本当の差は、たいてい「頑張り」ではなく「やり方」にあります。

本が10冊なら端から探しても一瞬。でも10万冊を毎回端から探していたら、一生かけても終わりません。手順が賢いか、ダサいか。その差は、数が増えるほど天文学的に開いていく。本講で味わうのは、この「速さは根性ではなく段取りで決まる」という、ちょっと痛快な事実です。

第5講で、私たちは「その問題は計算で解けるのか？」という問いを掘り下げました。 停止性問題のように原理的に解けない問題もある一方で、多くの問題は「時間さえかければ解ける」ものでした。 しかし、ここに落とし穴があります。「いつかは解ける」と「現実的な時間で解ける」は、まったくの別物なのです。 理論上は解けても、答えが出るのが100年後では、何の役にも立ちません。

そこで本講では、「解けるか」の次の問い――「どれくらい速く解けるか」に踏み込みます。 鍵を握るのは2つの概念です。問題を解くための手順そのものを指すアルゴリズムと、 その手順がデータ量に対してどれだけ手間を食うかを測る計算量。 この2つを身につけると、「なぜGoogleマップは一瞬で最短ルートを出せるのか」 「なぜ暗号は破られないのか」といった、現実の不思議の裏側が見えてきます。 プログラミングをしない人にとっても、コンピュータの"効き目"を理解する最重要の道具です。

もうひとつ、本講には大きなご利益があります。それは「ものの考え方」そのものが鍛えられること。 「この作業、データが増えたらどうなる？」「もっと無駄のない手順はないか？」―― こうした問いは、料理の段取りでも、書類整理でも、仕事の進め方でも役に立つ、普遍的な思考法です。 アルゴリズムと計算量は、コンピュータのための知識であると同時に、 「効率よく考える」ための一般教養でもあるのです。

そしてこの回は、これまでの講とこれからの講をつなぐ"結び目"でもあります。 第5講で学んだ「解けるか／解けないか」の地図の内側を、本講は「速く解けるか／事実上解けないか」で塗り分けました。 この計算量という物差しは、この先もずっと効いてきます。 第7講のデータ構造は「計算量を下げるための道具」ですし、第10講の暗号は「あえて計算量を爆発させて安全を守る」技術、 第15講のAIも「完璧な最適解をあきらめ、近似で折り合う」という、本講の発想の上に立っています。 「それは、どれくらいの手間で解けるのか？」という問いを持てるようになったあなたは、 もうコンピュータ科学の核心の一つを手にしています。

アルゴリズムとは何か → 探索で速さの差を体感 → ビッグオー記法 → 指数爆発 → ソート → P対NP。 数式は最小限にして、「手順の良し悪しが、どれほど結果を変えるか」を体で感じてもらいます。

アルゴリズムとは何か

まず捨てたい誤解

「アプリが重いのは、結局スマホが古いせいでしょ？」――半分は当たっています。でも、そこで買い替えに走る前に、知っておきたいことがあります。

同じ作業でも、手順（アルゴリズム）が下手だと、最新のマシンでも遅い。逆に手順が賢ければ、非力なマシンでも軽快に動く。速さは、馬力より段取りで決まる場面がとても多いのです。「アルゴリズム」と聞くと身構えますが、要は"段取りの良し悪し"の話だと思ってください。

アルゴリズムと聞くと身構えてしまいますが、要は「問題を解くための、きっちり決められた手順」のこと。 料理のレシピや、折り紙の折り方、目的地までの道順の説明も、広い意味ではアルゴリズムです。 ポイントは、誰がやっても（あるいは機械がやっても）同じ結果にたどり着くように、 手順が曖昧さなく定まっていること。 コンピュータは自分で考えてはくれないので、人間が「こうやって解け」と手順を与えてやる必要があります。 その手順がアルゴリズムです。

ごく簡単な例を挙げましょう。「テストの点数が並んだリストから、最高点を見つける」手順は、こう書けます。 ―― ①最初の人の点を「暫定1位」として覚える。②次の人から順に見ていき、暫定1位より高ければ、その点で覚え直す。 ③最後まで見たら、覚えている点が最高点。 当たり前すぎる手順ですが、これも立派なアルゴリズムで、計算量は O(n)（人数に比例）です。 私たちが無意識にやっている「全員ぶん見比べる」という作業を、曖昧さなく言葉にすると、こうなる。 アルゴリズムを考えるとは、頭の中の作業を、誰でも（機械でも）たどれる手順に翻訳することなのです。

良いアルゴリズムの条件

①正しい答えが出る ②必ず終わる、という大前提に加えて、 ③速い（手間が少ない） ④省メモリ（場所を食わない）であるほど良い。 同じ問題を正しく解くアルゴリズムは何通りもあり、その"出来"の差が、本講の主役である。

面白いのは、コンピュータがなくてもアルゴリズムは考えられる、という点です。 実際、アルゴリズムという考え方は、計算機が生まれるはるか前から数学のなかにありました。 「アルゴリズム（algorithm）」という言葉そのものが、 9世紀ペルシアの数学者アル＝フワーリズミーの名前に由来します。 彼が著した、方程式の解き方を体系的に述べた書物が、後にヨーロッパで「手順による計算法」の代名詞となり、 その名がなまって algorithm になりました。ちなみに「代数（algebra）」も、彼の著書の題名に由来します。 アルゴリズムは、コンピュータの技術ではなく、人類が積み上げてきた"考え方"の遺産なのです。

アルゴリズム研究の巨人・クヌース。 現代のアルゴリズム研究を語るうえで欠かせないのが、ドナルド・クヌースです。 彼が半世紀以上かけて書き継いでいる『The Art of Computer Programming（TAOCP）』は、 アルゴリズムの聖典とも呼ばれる大著で、いまも未完のまま執筆が続いています。 クヌースは「自分の本の組版が気に入らない」という理由で、 数式まで美しく組める組版ソフトTeX（テフ）を自作してしまったことでも有名です。 さらに、自著の誤りを見つけた人に「16進数で1ドル（約2.56ドル）の小切手」を贈るという、 ユーモアあふれる賞金制度でも知られています。 アルゴリズムの世界には、こうした探究心とこだわりの強い"職人"たちが、確かな土台を築いてきました。

探索で体感する、速さの差

雑なたとえ

探し方のうまい・へたを、紙の辞書で考えてみましょう。「algorithm」を引くのに、1ページ目から順にめくっていく人と、真ん中を開いて「あ、もっと後ろだな」と一気に半分に絞る人がいます。

前者が線形探索、後者が二分探索です。10ページなら大差ないのに、ページ数が増えるほど差は残酷に開く。100万ページでも、後者ならたった20回ほどで目的地に着きます。同じ「探す」でも、やり方ひとつでここまで違う――この体感が、計算量の話のいい入口になります。

「手順の良し悪しで、こんなにも違うのか」を、いちばん分かりやすく体感できるのが「探索（さがす）」です。 分厚い英和辞典（1万語が、すでにアルファベット順に並んでいるとします）から 「computer」という単語を探す場面を考えましょう。やり方は大きく2つあります。

① 線形探索（しらみつぶし）。1ページ目から順に、1語ずつめくって確認していく方法です。 運が悪ければ、最後の1語にたどり着くまで1万回めくることになります。 確実ですが、データが増えるほど比例して時間がかかります。

② 二分探索（半分ずつ捨てる）。すでに順番に並んでいることを活かします。 まず辞典のちょうど真ん中を開く。目当ての語が、開いたページより前か後かを見て、 いらない半分をまるごと捨てる。残った半分の、また真ん中を開く……これを繰り返します。

この二分探索、どこかで見た覚えはないでしょうか。そう、第1講の「数当てゲーム」とまったく同じ理屈です。 1回ごとに候補が半分になるので、1万語でも log₂10000 ≒ 14回 で見つかってしまう。 ―― 1万回 対 14回。これが「手順（アルゴリズム）」の差です。 しかもデータが増えるほど、この差は天文学的に広がります。 100万語に増えても、線形探索が最悪100万回かかるのに対し、二分探索はわずか約20回。 「並べ替えておく」というひと手間が、後の探索を桁違いに速くするのです。

具体的に、1〜100の中から「73」を二分探索で探す様子を追ってみましょう。 毎回、残った範囲のちょうど真ん中と比べ、いらない半分を捨てていきます。

範囲 1〜100 → 真ん中50。73は大きい → 51〜100に絞る
範囲 51〜100 → 真ん中75。73は小さい → 51〜74に絞る
範囲 51〜74 → 真ん中62。73は大きい → 63〜74に絞る
範囲 63〜74 → 真ん中68 → … と続け、数回で73に到達

100個の候補が、わずか7回ほどで1個に絞り込まれます（2⁷＝128 ≧ 100だから）。 「毎回半分」という、ただそれだけの工夫が、これほどの威力を発揮するのです。

大事な前提 二分探索が使えるのは「データが並んでいる」場合だけです。バラバラのデータでは真ん中を見ても判断できません。 だから、あらかじめ並べ替える「ソート」（後述）が重要になり、 探索とソートはセットで語られるのです。

計算量とビッグオー記法

「14回」「1万回」といった具体的な回数は、データの量によって変わります。 そこでコンピュータ科学では、具体的な回数そのものより、 「データの量 n が増えたとき、手間がどんな"勢い"で増えるか」に注目します。 この増え方の型を表す記法がビッグオー記法（O記法）です。 細かい係数や定数は無視して、伸びの本質だけを取り出すのが特徴です。

たとえば線形探索は、データが10倍になれば手間も約10倍。これを O(n) と書きます。 二分探索は、データが10倍になってもほとんど増えない。これを O(log n)。 次の表で、代表的な計算量を「遅い順」に並べてみましょう。 この表のイメージが頭に入っていれば、もう計算量の話の半分は理解したも同然です。

表1：代表的な計算量（n＝データ量。下にいくほど急激に遅くなる）
記法呼び名nが10倍になると…身近な例
O(1)定数時間変わらない（最強）配列の番地アクセス、ハッシュ表の参照（第7講）
O(log n)対数時間ほぼ変わらない二分探索
O(n)線形時間約10倍線形探索、合計を求める
O(n log n)準線形約13倍高速なソート（マージソート等）
O(n²)二乗時間約100倍素朴なソート、総当たりの組合せ比較
O(2ⁿ)指数時間天文学的（事実上不能）全部の組み合わせを総当たり

ここで強調したいのは、データが少ないうちは、計算量の差はほとんど気にならないということ。 n が10や20なら、O(n) でも O(n²) でも一瞬です。 差が牙をむくのは、n が大きくなったとき。 だから「少ないデータでは速かったプログラムが、本番の大量データで突然固まる」という事故が起きます。 プログラムの実力は、データが大きくなったときにこそ問われるのです。

ここで、第5講とつなげて大事な線引きをしておきましょう。 第5講では「解ける／解けない（計算可能か）」を区別しました。本講はその先で、 解ける問題のなかを「現実的に解ける／事実上解けない」でさらに分けています。 その境界線として、コンピュータ科学では O(n) や O(n²) のようにn の"べき乗"でおさまる計算量を「多項式時間」と呼び、 ざっくり"実用的に解ける"の目安とします。 一方 O(2ⁿ) のような指数時間は"事実上解けない"側。 じつは、さきほどのP対NPの「P」とは、まさにこの多項式時間で解ける問題のグループのことなのです。 「速さの型」が、問題の難しさを分類する物差しそのものになっている、というわけです。

どれくらい違うのか、具体的な数字で見てみましょう。 データが100万件（n＝10⁶）あるとき、1回の処理を1マイクロ秒（100万分の1秒）でこなせる機械を仮定します。 O(n) なら100万回で約1秒。O(n log n) なら約2千万回で約20秒。 ところが O(n²) だと1兆回で、なんと約11日かかります。 さらに O(2ⁿ) となると、n＝100でも宇宙の年齢をはるかに超える。 同じ問題・同じ機械でも、アルゴリズムの型が違うだけで「1秒」と「11日」と「永遠」に分かれるのです。 速さを決めているのは、機械のパワーよりも、まず手順の良し悪し――この感覚が、本講のいちばんの収穫です。

指数爆発の恐怖

説明役「指数爆発」のいちばん身近な例、たぶん幹事をやったことがある人なら知ってます。飲み会の日程調整です。

聞き役あー……参加者が増えると、急に「全員OKな日」が消滅するやつ。

説明役そう。候補日と人数が少し増えるだけで、組み合わせが爆発的にふくれ上がる。コンピュータですら音を上げるのが、この「ちょっと増えただけで手に負えなくなる」タイプの問題なんです。

表のいちばん下、O(2ⁿ) の恐ろしさを、もう少し味わっておきましょう。 これは「n が1増えるだけで、手間が2倍になる」という増え方です。 n が10で約1000、20で約100万、30で約10億、60ともなれば、 その数は、最速のスーパーコンピュータを宇宙の年齢ぶん走らせても終わらないほどになります。 「総当たりで全部の組み合わせを試す」という素朴な作戦が、なぜしばしば破滅するのか。理由はこれです。

指数爆発の怖さを伝える、古い寓話があります。 チェス盤を発明した男が、王様にこう願い出ました。 「1マス目に米1粒、2マス目に2粒、3マス目に4粒……と倍々で、64マスぶんの米をください」。 王様は「なんとつつましい願いだ」と笑って許しました。 ところが計算してみると、最後のマスでは 2⁶³ 粒、全マスの合計は約1844京粒―― 全世界が何百年生産しても、到底足りない量でした。 「倍々」を甘く見ると破滅する。指数関数とは、それほどまでに凶暴なのです。 コンピュータ科学者が O(2ⁿ) のアルゴリズムを「使い物にならない」と嫌うのも、同じ理由です。

だからこそ、賢いアルゴリズムを考える価値があります。 総当たりなら O(2ⁿ) で絶望的な問題でも、 うまい工夫で O(n²) や O(n log n) に落とせれば、現実的な時間で解けるようになる。 「コンピュータが速いから解ける」のではなく、「賢い手順を見つけたから解ける」。 ハードの進歩より、アルゴリズムの工夫のほうが、効き目が大きいことも珍しくないのです。

ところで、この指数爆発は「困りもの」であると同時に、頼もしい味方にもなります。 パスワードの総当たり攻撃を思い出してください（第1講）。 パスワードを1文字長くする、使う文字種を増やすたびに、試すべき組み合わせは指数的に膨れ上がります。 攻撃する側にとっては、まさに「米粒が宇宙規模に増える」あの爆発が立ちはだかる。 解く側を絶望させる指数爆発を、守る側があえて利用する――これが暗号の基本戦略です（第10講）。 同じ「指数関数の凶暴さ」が、攻撃者には壁、防御者には盾になる。 計算量の感覚を持つと、こうした攻防の構図までスッと見通せるようになります。

並べ替えの奥深さ ― ソートアルゴリズム

計算量の違いを体感する定番の題材が、データの並べ替え（ソート）です。 トランプを小さい順に並べる方法を考えると、手順の違いがよく分かります。

  • バブルソート：隣どうしを比べ、逆順なら入れ替える。これを端から何度も繰り返す。 素朴で分かりやすいが、O(n²) と遅い。トランプ全部を、隣同士の交換だけで並べるイメージ。
  • マージソート：山を半分ずつに分割していき、最小単位まで分けてから、 「整列済みの2つの山を合体させる」操作で組み上げる。O(n log n) と速い。
  • クイックソート：基準値を1つ選び、それより小さい組・大きい組に振り分け、各組をまた同じやり方で処理する。 平均 O(n log n) で、実用上とても高速。

「分けて、それぞれを片付けて、まとめる」というマージソートやクイックソートの戦略は、 分割統治法と呼ばれ、アルゴリズム設計の王道のひとつです。 大きな問題を、自分と同じ形の小さな問題に分けて解く――この発想は、いろいろな場面で顔を出します。

身近な例で考えてみましょう。本棚のマンガを巻数順に並べ替えるとき、 多くの人は、近くにある巻どうしを少しずつ入れ替えていきます。これはバブルソートに近いやり方で、 冊数が増えるほど手間が O(n²) で膨らみ、つらくなる。 一方、いったん「1〜10巻の山」「11〜20巻の山」とおおまかに分けてから各山を並べ、最後に合体させれば、 ずっと効率よく片付きます。これが分割統治の発想です。 同じ「並べ替え」でも、やり方しだいで労力がまるで違う―― これは、コンピュータに限らず、私たちの日常の作業にも通じる教訓です。 ちなみに、実際のプログラミング言語に組み込まれたソート機能は、 マージソートやクイックソートなど複数の手法を、データの性質に応じて賢く組み合わせた 高性能な実装になっていることが多く、私たちは普段その恩恵を意識せず使っています。

そして、ここでも第1講のエントロピーが顔を出します。 じつは「比較によって並べ替える」やり方は、理論上 O(n log n) より速くはできない、と証明されています。 なぜか。n個のものの並び順は n!（n の階乗）通りあり、 そのどれか1つに絞り込むには、第1講の「Yes/No質問」が最低でも約 n log n 回必要だからです。 並べ替えの速さの限界すら、情報量で決まっている。 第1講で学んだ「1つに絞るのに必要な質問の回数」が、ここで効いてくるのです。

P対NP問題 ― 100万ドルの難問

聞き役「答えの確認は一瞬なのに、答えを見つけるのは絶望的に大変」って、ちょっとずるくないですか。

説明役でも身に覚えありませんか? ジグソーパズルって、完成形を見せられたら「合ってる」と一瞬で分かる。でも自力で完成させるのは、めちゃくちゃ時間がかかる。

聞き役たしかに。「答え合わせは楽、答え探しは地獄」。それが世界中の難問の正体だと言われると、妙に納得しますね。

最後に、コンピュータ科学で最も有名な未解決問題に触れておきましょう。 まず、問題を2つのタイプに分けて考えます。

  • P：答えをそこそこ速く（現実的な時間で）見つけられる問題。
  • NP：答えを見つけるのは大変かもしれないが、 答えを見せられれば、それが正しいかは速く確認できる問題。

分かりやすいのが数独（ナンプレ）です。完成したマス目を見せられれば、 「ルール通りに埋まっているか」はすぐ確認できます（これがNPの性質）。 でも、白紙の状態から解くのは、ぐっと難しい。 ジグソーパズルも同じで、完成品が正しいかの確認は一瞬でも、ゼロから組むのは大変です。

ここで、ひとつの根源的な問いが立ちます。 「答えを速く"確認"できる問題は、答えを速く"発見"することもできるのか？」 言い換えれば、P と NP は同じものなのか（P＝NP）、それとも違うのか（P≠NP）。 これが、クレイ数学研究所が賞金100万ドルをかけた7つの難問のひとつ、P対NP問題です。

NPの代表として有名なのが巡回セールスマン問題です。 「複数の街をすべて1度ずつ訪れて出発点に戻る、最短の道順を見つけよ」という問題で、 一見やさしそうですが、街の数が増えると経路の組み合わせが指数的に爆発し、総当たりでは手に負えなくなります。 ところが「このルートの総距離は◯◯kmだ」と答えを見せられれば、検算はあっという間。 まさに「発見は難しいが、確認は簡単」なNPの典型です。 配送ルートの最適化や、基板の配線設計など、現実の重要な問題の多くが、この巡回セールスマン問題の仲間に属しています。 だからこそ、完璧な最短解を探すのをあきらめ、「そこそこ良い解」で手を打つ近似アルゴリズムが、実務では大活躍するのです。

多くの研究者は「P≠NP（確認は楽でも、発見は本質的に難しい）」だろうと予想していますが、 いまだに誰も証明できていません。 そして、これは机上の空論ではありません。もし万一 P＝NP が証明されてしまったら―― 現代の暗号（第10講）の多くは「解読の答えを見つけるのが事実上不可能」という前提に立っているので、 その前提が崩れ、世界中のセキュリティが根底から揺らぐことになります。 抽象的な数学の問いが、私たちの暮らしの安全と、じかにつながっているのです。

現代への応用

Googleマップは、なぜ一瞬で最短ルートを出せるのか

カーナビや乗換案内が、膨大な道路網からほぼ一瞬で最短ルートを出せるのは、 賢いアルゴリズムのおかげです。 もし「考えられる全ルートを総当たりで比較」していたら、それこそ指数爆発で永遠に終わりません。 実際には、出発点に近いところから順に最短距離を確定していく賢い手順（ダイクストラ法や、その改良版）を使い、 無駄な計算をばっさり省いています。 この「点と線のネットワークを効率よくたどる」話は、次の第8講（グラフ）でくわしく扱います。 日常の便利さの裏には、必ず計算量と戦うアルゴリズムがいるのです。

「速さ」がサービスの価値を決める

検索エンジンが膨大なWebから一瞬で結果を返すのも、 通販サイトが何百万の商品から候補を絞るのも、SNSがタイムラインを組むのも、 すべて計算量との戦いです。 同じ機能でも、アルゴリズムが非効率なら「重くて使えないサービス」になり、 効率的なら「快適なサービス」になる。 ユーザー体験の良し悪しは、見えないところでアルゴリズムが決めているのです。 そして暗号（第10講）のように、「わざと"発見"を難しくする」ことを安全の根拠にする技術もある。 計算量は、速さの追求にも、安全の確保にも使える、両刃の剣なのです。

「完璧」をあきらめる、という戦略

本講のもうひとつの大事な教訓は、「完璧な答えを、いさぎよくあきらめる」という発想です。 巡回セールスマン問題のように、厳密な最適解を求めると指数爆発してしまう問題は、現実にたくさんあります。 そこで実務では、「最適ではないが、十分に良い解」を高速に求める近似アルゴリズムや、 運まかせの要素を取り入れて答えに近づく手法が活躍します。 配送計画も、AIの学習も、ゲームの思考ルーチンも、多くは「完璧」ではなく「実用上"いい感じ"」を狙っています。 解けない問題と正面衝突せず、賢く折り合いをつける――これも、計算量を知った者だけができる戦い方です。 この「ほどほどで手を打つ」知恵は、第15講のAIにもつながっていきます。

よくある誤解

誤解①：「コンピュータが速いから、どんな処理も一瞬で終わる」

アルゴリズムの計算量しだいです。O(2ⁿ) のような手順だと、どんな高速マシンでもデータが少し増えただけで終わらなくなります。 速さを決めるのはハードだけでなく、手順そのものです。

誤解②：「計算量が小さいアルゴリズムが、つねに最善」

データが少ないうちは、計算量の大きい単純な手順のほうが速いこともあります。 O記法は「nが大きいときの傾向」の話。実際には、データ量や状況に応じた使い分けが必要です。

誤解③：「最新のハードを買えば、遅いプログラムも速くなる」

焼け石に水のことが多いです。O(n²) を O(n log n) に変えるアルゴリズムの改善のほうが、 ハードの買い替えよりはるかに効きます。本質的な遅さは、手順を変えないと直りません。

誤解④：「P対NPは、ただの数学の遊び」

現実に直結しています。多くの暗号は「答えの発見が難しい（P≠NP的）」ことを安全の根拠にしているため、 もしP＝NPなら、世界中の暗号やセキュリティの前提が崩れます。

誤解⑤：「O記法の数字が同じなら、速さも同じ」

O記法は「増え方の型」だけを表し、定数倍の違いは無視します。 同じ O(n log n) でも、実装によって実際の速さは数倍違うことがあります。 O記法は"大づかみの傾向"をつかむ道具であって、最後の細かいチューニングは別問題、と心得ましょう。

この発想、生活にも出ます

「頑張る量を増やす」より「やり方を変える」ほうが効く――計算量の話は、そのまま日々の段取りの教訓です。散らかった部屋を片づけるのに毎回全部の箱を開けて探す人と、ラベルを貼って分類しておく人。後者は、探し物という"処理"が桁違いに速い。

残業を増やす前に手順を疑う。気合を入れる前に並べ方を変える。規模が大きくなるほど、根性より段取りがものを言う。これは仕事でも家事でも、けっこう冷酷なまでに当てはまる法則です。

まとめ

この講で持ち帰ってほしいのは、ビッグオー記法の暗記ではありません。「遅い」を、根性不足ではなく"手順の問題"として見直せる目です。これが身につくと、何かが重いとき、買い替える前にまず「やり方、変えられないか?」と問えるようになります。

  • 同じ問題でも、アルゴリズム次第で速さは桁違い（線形探索1万回 対 二分探索14回）。
  • 計算量はビッグオー O( ) で「データ増加に対する手間の伸び方」を表す。O(2ⁿ) は事実上解けない。
  • 指数爆発は凶暴。賢い手順で O(n²) や O(n log n) に落とせれば、現実的な時間で解ける。
  • 比較ソートの速さの下限 O(n log n) も、情報量（第1講のエントロピー）で説明できる。
  • P対NP：答えの「確認」が楽な問題で「発見」も楽か、は未解決の100万ドル難問。暗号の安全性と直結。

考えてみよう

  1. ソート済み100万件のデータから1件を二分探索で探すと、最悪およそ何回の比較で済むか（log₂(10⁶) を見積もる）。
  2. O(n²) のアルゴリズムで、n が10倍になると手間は何倍になるか。
  3. 「少ないデータでは速かったのに、本番で突然固まった」プログラムは、どんな計算量だった可能性が高いか。
  4. 「ジグソーパズルを完成させる」のは P と NP のどちらの感覚に近いか。完成品の検証と、ゼロから組む難しさを比べて考えよ。
  5. 遅いプログラムを速くしたい。「高性能なPCに買い替える」と「アルゴリズムを見直す」、本質的に効くのはどちらか。
ヒントと答え

(1) log₂(10⁶) ≈ 20 回。100万件でも、たった20回前後で特定できる。
(2) 10² ＝ 100倍。二乗時間はデータ増加にとても弱い。
(3) O(n²) や O(2ⁿ) など、データ量に弱い計算量。少量では一瞬でも、大量データで急に破綻する。
(4) 完成品が正しいかの確認は速いが、ゼロから組むのは組合せ的に大変――NPの直感そのもの。
(5) 多くの場合アルゴリズムの見直し。計算量の型そのものを改善するほうが、ハードの数倍程度の高速化よりはるかに効く。
